# FreeRTOS / Zephyr Labs

A hands-on embedded systems lab series focused on building practical intuition for **real-time operating systems**, **low-level firmware design**, and **hardware/software integration** on microcontrollers.

This repo explores the core patterns used in RTOS-based embedded systems: task scheduling, deterministic timing, inter-task communication, interrupt handling, shared resource protection, peripheral integration, and system observability. The goal is not just to use FreeRTOS primitives, but to understand how they support clean, modular, and production-minded firmware architecture.

## What this repo demonstrates

- RTOS task creation, scheduling, and priority-based execution
- Deterministic periodic task design
- Safe data transfer between tasks using queues
- Event-driven firmware using semaphores and task notifications
- Shared peripheral protection using mutexes
- Correct interrupt-to-task integration with RTOS APIs
- Peripheral driver integration under an RTOS
- Debug, reliability, and system health instrumentation
- Modular task-based architecture for embedded systems

## Low-Level Driver Work

This repo includes a register-level SPI2 driver for the STM32F401RE and an ADXL345 accelerometer driver built on top of it.

Driver files:
- `common/drivers/spi.c` — STM32 SPI2 initialization, GPIO alternate-function setup, chip-select control, blocking byte/multi-byte SPI transfers
- `common/drivers/adxl345.c` — ADXL345 register reads/writes, device ID check, measurement mode setup, range configuration, multi-byte XYZ sample reads

This demonstrates:
- Direct CMSIS/register-level peripheral configuration
- GPIO alternate-function setup for SPI pins
- SPI mode, baud rate, frame format, and NSS/software chip-select configuration
- Sensor driver layering over a reusable bus driver
- RTOS integration through Lab 10


## Lab Breakdown

### Lab 1 — Multiple Tasks and Priorities
Introduces the FreeRTOS scheduler through multiple concurrent tasks running at different priorities. Tasks perform visible behaviors such as UART output or timed GPIO activity to demonstrate **preemption**, **priority effects**, and **round-robin execution** among equal-priority tasks.

**Key concepts:** task creation, scheduling behavior, priorities, task structure, stack sizing

### Lab 2 — Periodic Tasks and Deterministic Timing
Builds periodic tasks that execute at fixed rates using `vTaskDelayUntil()`. This lab focuses on **timing discipline** and shows how RTOS-based systems maintain regular execution without relying on busy-wait loops or imprecise delay patterns.

**Key concepts:** periodic execution, deterministic timing, blocking delays, timing discipline

### Lab 3 — Queues for Inter-Task Communication
Implements producer/consumer task patterns using FreeRTOS queues. Data is generated by one task and safely consumed by another, demonstrating a cleaner alternative to shared globals and reinforcing structured communication between software components.

**Key concepts:** queues, producer/consumer design, task communication, synchronization, data ownership

### Lab 4 — Binary Semaphores and Event Signaling
Demonstrates event-driven task design using binary semaphores. A task blocks until signaled by another task, a timer callback, or an external event, showing the difference between **data transfer** and **wake-up signaling**.

**Key concepts:** binary semaphores, event signaling, blocked task behavior, CPU-efficient design

### Lab 5 — Mutexes and Shared Resource Protection
Explores how multiple tasks safely access a shared resource such as UART. This lab demonstrates why mutual exclusion matters and introduces the foundations of **resource protection** and **priority inversion awareness**.

**Key concepts:** mutexes, shared resource protection, mutual exclusion, priority inversion fundamentals

### Lab 6 — Task Notifications as Lightweight Signals
Uses **direct-to-task notifications** as a lighter-weight alternative to queues and semaphores for simple signaling. This lab highlights awareness of efficient FreeRTOS-specific primitives commonly used in embedded systems.

**Key concepts:** task notifications, lightweight signaling, efficient task wake-up, primitive selection

### Lab 7 — Software Timers and Deferred Actions
Introduces one-shot and periodic software timers, with callbacks used to signal tasks rather than perform heavy work directly. This demonstrates a cleaner separation between **timing events** and **application logic**.

**Key concepts:** software timers, one-shot timers, periodic timers, deferred work patterns

### Lab 8 — Interrupts with RTOS
Connects hardware interrupts to RTOS task execution using the correct `FromISR` APIs. An ISR signals a task, and the task performs the heavier processing outside interrupt context, reflecting a common pattern in real embedded firmware.

**Key concepts:** ISR/task boundaries, `FromISR` APIs, interrupt-driven task wake-up, RTOS interrupt integration

### Lab 9 — UART Command Interface / Mini CLI
Builds a simple UART-based command interface for runtime interaction and debugging. Commands such as `help`, `stats`, `led on`, or `tasks` provide a practical example of embedded debug tooling and coordinated task-driven behavior.

**Key concepts:** UART command handling, parsing, embedded debug tooling, coordinated task actions

### Lab 10 — Low-Level Driver design and integration
I wrote a low-level driver in C for the ADXL345 accelerometer. I integrated this with an RTOS-based application. This lab focuses on **driver/application layering** and demonstrates relevance beyond scheduler-only demos.

**Key concepts:** RTOS + peripheral integration, modular driver structure, acquisition tasks, processing tasks

### Lab 11 — System Health, Debug, and Reliability
Adds observability and production-minded diagnostics, such as stack high-water marks, heap usage checks, runtime statistics, watchdog concepts, and fault handling paths. This lab emphasizes reliability and awareness of failure modes in embedded systems.

**Key concepts:** observability, runtime diagnostics, stack/heap monitoring, watchdog thinking, robustness

### Lab 12 — Integrated Mini Application
Combines earlier RTOS concepts into a cohesive embedded application. Example building blocks include sensor acquisition, processing, communications, interrupt-driven events, shared resource protection, and health reporting, all organized using clear task boundaries.

**Key concepts:** embedded architecture, task decomposition, integrated RTOS design, modular firmware structure

## Why this repo exists

Many RTOS examples stop at simple API usage. This repo is meant to go further by building intuition for how embedded firmware is actually structured when multiple tasks, peripherals, interrupts, and system constraints interact at once.

The long-term focus is on writing firmware that is:

- deterministic
- modular
- hardware-aware
- debuggable
- scalable beyond toy examples

This repo is being developed as a practical embedded systems portfolio project to strengthen and demonstrate skills relevant to:
